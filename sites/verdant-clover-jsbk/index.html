System Architecture

Hardware

  • Lenovo laptop — Ubuntu 24.04 server
  • Lid-closed operation (AC power)
  • On-site at La Retama shop
  • Wi-Fi auto-reconnect configured

Remote Access

  • Tailscale — dedicated client tailnet
  • SSH key-only, UFW tailnet-bound
  • Portainer (Docker management)
  • Fail2ban, auto-updates

Agent Stack

  • Hermes Agent v0.16.0
  • Python 3.11 venv, unprivileged "hermes" user
  • Docker sandbox terminal backend
  • systemd user service (boot persistent)

AI Models

  • Primary: economical free-tier model
  • Delegate: stronger paid model (owner key)
  • TTS: Gemini voice (warm voice selected)
  • STT: on-device speech-to-text

Tools Enabled

  • DuckDuckGo web search
  • Gemini TTS (voice output)
  • Docker sandbox (code execution)
  • Read-only Odoo/ERP connector (built, parked)

Security Posture

  • Lynis hardening index: 76
  • No docker.sock, read-only mounts
  • Manual approval policy (fail-closed)
  • Env file mode 600, no key forwarding
Gateway pattern Claudia only knows users who have written to her in the current run — her contact directory rebuilds on restart. Proactive outbound messages to the owner must be triggered via the Telegram Bot API directly (not via the agent's own list). This is documented and worked around.

Ops & Maintenance Reference

Remote access

  • SSH via Tailscale (key auth only)
  • Portainer on tailnet port
  • No direct internet SSH exposure
  • Android → Termux/JuiceSSH

Agent service

  • systemd user service, lingering
  • Restarts automatically on crash
  • Contact dir rebuilds on restart
  • Check: journalctl --user -u hermes

Monitoring

  • healthchecks.io dead-man (5/10 min)
  • Telegram "DAV Infra Alerts" group
  • Email fallback alert
  • Brief internet outage = expected (self-recovers)

Key management

  • All keys in KeePassDX
  • Agent env file: mode 600
  • Owner keys: SETUP command (never seen by dev)
  • Bot token: KeePassDX entry
Internet outage protocol If "server down" alert fires: (1) wait 15 minutes — most outages are brief shop internet blips and the server self-recovers. (2) If alert persists: SSH in via Tailscale, check systemctl --user status hermes. (3) If SSH unreachable: physical visit or ask Yacub to reboot the laptop. The dead-man has a 10-min grace so short outages don't page.
Before any Odoo write access goes live Full staging + read-only phase required (minimum 2 weeks). Update SOUL.md approval gates. Test propose-then-confirm flow on staging Odoo before touching production. Get explicit owner sign-off.
